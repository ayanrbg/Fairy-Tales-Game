# Patch: полная загрузка сказки + все переводы текста

## Проблема

Клиент кэшировал только картинки и озвучку. **Текст сказки не кэшировался** — он
тянулся живьём (`GET /api/tales/:id?lang=`) при каждом открытии, только для текущего
языка. Из-за этого:

1. Офлайн текста нет.
2. При смене языка текст не перезагружался корректно.
3. Главный баг: для серверных сказок после `GET /tales/:id?lang=uz` клиент звал
   `POST /tales/:id/personalize` **без `lang`** → сервер брал русскую версию,
   персонализировал её и возвращал → узбекский текст затирался русским.

## Что уже сделано на клиенте

- При загрузке сказки клиент теперь качает текст **всех** языков
  (`ru/kz/en/uz`) и кладёт каждый в кэш (`tales-text/{id}/{lang}.json`).
- Экран читает текст из кэша для текущего языка, `{childName}` подставляет сам,
  `{m:...|f:...}` резолвит при отрисовке. Серверный `/personalize` для скачанных
  сказок больше не нужен.
- В `/personalize` теперь передаётся `?lang=`.

Чтобы это заработало корректно, нужны изменения на сервере ⬇️

---

## Что сделать на сервере

### 1. `GET /api/tales/:id?lang=X` — поле `lang` в ответе должно быть РЕАЛЬНЫМ языком

Сейчас, если у сказки нет перевода на `X`, сервер отдаёт версию по умолчанию (ru),
но в ответе важно вернуть **реальный** код языка отданной версии, а не эхо `X`.

```js
// псевдокод хендлера GET /api/tales/:id
const requested = req.query.lang;
const tale = loadTale(id, requested);          // может вернуть дефолтную (ru) версию
if (!tale) return res.status(404).json({ error: "Tale not found" });

res.json({
  id: tale.id,
  title: tale.title,
  lang: tale.lang,        // ⚠️ ВАЖНО: реальный язык отданной версии, НЕ requested
  totalPages: tale.totalPages,
  pages: tale.pages,      // СЫРЫЕ шаблоны (см. п.2)
  genderedPages: tale.genderedPages
});
```

**Почему критично:** клиент качает все 4 языка и кэширует версию только если
`response.lang === запрошенный`. Если сервер будет эхать `lang` = запрос, клиент
сохранит русский текст под ключом `uz` → офлайн узбек снова покажет русский.
С честным `lang` клиент просто не закэширует отсутствующий перевод (и это правильно).

### 2. `pages` в `GET /api/tales/:id` — это СЫРЫЕ шаблоны (без подстановки)

Поле `pages` должно содержать **неперсонализированный** текст с плейсхолдерами —
клиент подставляет их сам:

| Шаблон | Кто резолвит на клиенте |
|--------|--------------------------|
| `{childName}` | подстановка имени ребёнка |
| `{m:слово\|f:слово}` | выбор по полу при отрисовке (`PageNavigator`) |

Это и так должно выполняться (у `GET /tales/:id` нет `name`/`gender`, персонализировать
нечем). **Требование к данным:** в исходных текстах сказок имя должно стоять как
`{childName}`, а родовые формы — как `{m:...|f:...}`, в тех же местах, где их вставляет
`/personalize`. Тогда клиентская подстановка даёт идентичный результат во всех языках,
офлайн. Формат гендера — строго `{m:...|f:...}` (regex клиента: `\{m:([^|]*)\|f:([^}]*)\}`).

### 3. `POST /api/tales/:id/personalize?lang=X` — учитывать `lang`

Резервный путь (онлайн, для ещё не скачанных серверных сказок). Сейчас игнорирует язык.

```js
// было: const tale = loadTale(id);            // или хардкод 'ru'
const lang = req.query.lang;                    // читаем язык из query
const tale = loadTale(id, lang);                // тот же loadTale(id, lang), что в GET и narrate
if (!tale) return res.status(404).json({ error: "Tale not found" });
// ...дальше подстановка name/gender как раньше...
```

### 4. (Опционально) перечень доступных языков сказки

Не обязательно — клиент определяет наличие перевода по полю `lang` из п.1.
Но если удобно, можно добавить в `GET /api/tales` и `GET /api/tales/:id` массив:

```json
"langs": ["ru", "kz", "uz"]
```

Тогда клиент сможет качать только существующие переводы (меньше лишних запросов).

## Проверка

```
# 1. У сказки с переводом на uz — должен вернуть uz и lang:"uz"
GET /api/tales/<id>?lang=uz        → { "lang": "uz", "pages": ["{m:...|f:...}", ...] }

# 2. У сказки БЕЗ uz — fallback на ru, но lang честный:
GET /api/tales/<id_без_uz>?lang=uz → { "lang": "ru", ... }   # НЕ "uz"

# 3. personalize учитывает язык:
POST /api/tales/<id>/personalize?lang=uz  body {name,gender} → узбекские pages
```

Пункты **1 (честный lang) и 3 (lang в personalize)** — обязательны.
Пункт 2 (сырые шаблоны) — требование к данным, скорее всего уже выполнено.
Пункт 4 — опционально.
